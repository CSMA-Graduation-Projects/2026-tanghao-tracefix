# 毕业设计符合性检查报告

## 1. 项目总体判断

当前项目以 `cfix/` 为核心，包含 Vue3 前端、FastAPI 后端、MySQL 建表脚本、论文 Word 文档、答辩 PPT 和若干 README。代码结构较完整，能看出“代码生成 -> 执行反馈 -> 自动修复 -> 版本对比 -> 实验评估”的系统主线，论文和 PPT 也基本围绕该系统展开。主要短板集中在工程可复现性、测试支撑、部署展示和实验数据一致性：本地测试只有占位测试，`docker-compose.yml` 与 README 引用的 SQL 文件名和实际文件不一致，在线部署目前只能确认站点头信息可达但未能完整拉取页面内容，部分实验结果文件还存在最终通过率超过 100% 等明显口径问题。因此项目更接近“基本框架已具备，但还有较明显短板，需要补强”。

- 代码目录：`cfix/cfix_frontend/`、`cfix/cfix_server/`
- 数据库脚本：`cfix/cfix_db.sql`
- 论文文件：`cfix/paper/基于代码执行反馈的代码生成与自修复系统的设计与实现.docx`
- PPT 文件：`cfix/ppt/基于代码执行反馈的代码生成与自修复系统答辩.pptx`
- 说明文档：根目录 `README.md`、`cfix/cfix_frontend/README.md`、`cfix/cfix_server/README.md`
- 部署配置：根目录 `docker-compose.yml`
- 实验输出：`cfix/cfix_server/data/out/reg_qwen/`、`cfix/cfix_server/data/out/reg_qwen_file/`、`cfix/cfix_server/data/out/exp_cmp_file/`
- 测试目录：`cfix/cfix_server/tests/test_placeholder.py`

## 3. 项目主要优点

1. 代码不是单纯页面壳子，后端包含认证、任务、运行、版本、实验、数据集、模型配置、沙箱执行等模块。
2. 前端 `WorkBench.vue`、历史任务、数据集、实验评估、系统设置等页面较完整，能支撑答辩展示主流程。
3. 数据库设计覆盖用户、会话、任务、版本、用例、执行记录、trace、plan、lesson、实验等核心对象。
4. 论文结构完整，包含需求分析、系统设计、系统实现、系统测试、应用场景、总结展望等章节，并嵌入较多图表和截图。
5. PPT 16 页，整体逻辑清楚，主题与论文和代码中的核心功能基本一致。

## 4. 主要问题

1. 部署配置存在明显错误：`docker-compose.yml` 和根 README 引用 `cfix/cfix_schema.sql`，但本地实际文件是 `cfix/cfix_db.sql`，会导致一键 Compose 初始化数据库失败。
2. 测试支撑不足：后端测试目录只有 `test_placeholder.py`，内容为 `assert True`；`req.txt` 未包含 pytest，本地执行 `python -m pytest cfix/cfix_server/tests` 失败，原因是当前环境无 pytest 模块。
3. 实验数据口径需要核验：`data/out/reg_qwen/reg_qwen.md` 中出现“最终通过率：200.00%”这类不合理结果；多个结果文件显示 `trace 非空题目数：0`，与论文/PPT强调 Trace 辅助修复的叙述需要补充解释。
4. 在线部署展示未能完整验证：HTTP HEAD 可确认 EdgeOne 站点存在，携带 cookie 后根路径返回 `200 OK`，但当前环境中 GET 页面内容失败；未能确认登录、任务创建、自动修复、实验页是否可演示。
5. 论文和 PPT 中部分函数/模块称呼与代码不完全一致，例如 PPT 写 `task_svc.auto_repair()`、`run_svc.execute()`，而代码中主流程是 `TaskService.auto_fix()`、`RunService.run_version()`，答辩时容易被追问。
6. README/部署说明缺少完整的归档级操作闭环，例如默认演示账号、真实模型配置说明、线上部署后端 API 地址、失败排查步骤、演示脚本。
7. 后端保留 `stream` placeholder 路由，若 PPT 或论文暗示实时流式任务进度，需要避免夸大。

## 5. 代码与系统实现检查结果

### 已有基础

- 前端基于 Vue3、Vite、Element Plus、Pinia、Vue Router、Axios，包含登录、工作台、历史任务、数据集、实验评估和设置页。
- 后端基于 FastAPI、SQLAlchemy、MySQL，代码分层包含 `api/v1`、`svc`、`agent`、`sand`、`llm`、`models`、`repo`、`schemas`、`data`、`utils`。
- 核心服务能对应论文主线：`task_svc.py` 负责任务和自动修复编排，`run_svc.py` 负责执行测试和反馈整理，`trace_svc.py`、`plan_svc.py`、`fix_svc.py`、`lesson_svc.py`、`rb_svc.py` 对应 trace、修复计划、修复代码、经验记录和回滚控制。
- 数据库脚本有 303 行，覆盖主要业务表；后端 app Python 文件约 93 个，前端 src 下 Vue/JS 文件约 31 个，规模基本符合毕业设计系统项目。
- `data/out/` 下保留了部分实验输出和 Markdown/CSV/JSON 结果，可作为论文实验的部分支撑材料。

### 存在问题

- `docker-compose.yml` 挂载 `./cfix/cfix_schema.sql`，但该文件不存在；实际文件为 `cfix/cfix_db.sql`。
- 测试目录只有占位测试，缺少对登录、任务创建、用例解析、执行反馈、自动修复、回滚、实验统计的真实测试。
- `stream.py` 返回固定 placeholder 数据，说明实时任务流功能未完整实现。
- 真实模型调用依赖 `.env` 中的 provider、base_url、api_key 等配置，当前材料不足以判断答辩现场是否能稳定调用真实模型。
- safe_exec 和 docker 沙箱都存在，但 Compose 默认 `SANDBOX_BACKEND=safe_exec`，论文/PPT强调 Docker 隔离时需说明本地演示与强隔离部署的差异。
- `reg_qwen.md` 中 200% 通过率是明显统计或输出错误，需要立即修正。

### 建议补充

- 修正 Compose 和 README 中 SQL 文件名，保证 `docker compose up --build` 至少能完成数据库初始化。
- 增加真实测试：至少覆盖后端 API smoke、用例规范化、`run_version`、`auto_fix` 基本流程、实验报告统计。
- 在 README 中补充“最小可演示流程”：登录、创建任务、生成代码、运行测试、自动修复、查看版本、导出报告。
- 对实验脚本输出做口径校验，禁止出现通过率超过 100%、trace 指标与论文叙述不一致却无说明的情况。
- 如果不使用 SSE 流式进度，应在论文/PPT中不要强调该能力；如要展示，需要把 `stream.py` 从 placeholder 补成真实进度接口。

## 6. 论文检查结果

### 与系统一致的部分

- 论文围绕“基于代码执行反馈的代码生成与自修复系统”展开，不是泛泛写管理系统。
- 论文提到的前端页面、后端分层、数据库表、任务/版本/运行记录/trace/plan/lesson/实验评估，大多能在本地代码中找到对应实现。
- 论文第 5 章对 `WorkBench`、`task_svc`、`run_svc`、`fix_svc`、`bench_bank.py` 等实际模块有描述，说明不是完全脱离代码的概念性论文。
- 论文包含较多图片和表格，docx 中识别到 37 个媒体文件和 52 个表格，形式上具备系统展示材料。

### 可疑或偏空泛的部分

- 第 6 章测试章节写得较完整，但本地代码测试支撑不足，当前只有占位测试，难以支撑“单元测试、集成测试、系统测试、性能测试”都已充分完成的叙述。
- 实验章节提到函数级、类文件级、文件级三类实验，并给出 90%/100% 等结论；本地可直接检查到的原始输出主要集中在 `file_ultra` 以及一个 5 题回归报告，材料不足以完整证明论文中三类实验全部结论。
- `reg_qwen.md` 的 200% 最终通过率与论文实验可信度冲突。
- 论文多次强调 Trace、Lesson、Rollback 对修复的辅助作用，但部分实验输出显示 `trace 非空题目数：0`，需要解释是 trace 未触发、未保存，还是统计字段未接入。
- 部署相关内容较少，关键词命中很低，缺少线上部署访问、后端 API、环境变量、演示账号等归档说明。

### 建议如何修改

- 在论文测试章节中补充与本地测试文件对应的真实测试用例和执行截图，不要只放界面截图或泛化结论。
- 修正所有实验统计口径，附上样本数、数据集名称、模型配置、最大轮次、是否启用 Trace/Lesson/Rollback、原始结果文件路径。
- 对“Trace/Lesson/Rollback 有提升”的结论改成有条件表述，并说明文件级实验中完整链路未必优于错误反馈多轮修复。
- 增加“部署与运行说明”小节，说明本地 Compose、后端启动、前端构建、线上地址、演示账号、模型配置要求。
- 统一代码函数名和论文术语，例如按代码实际名称写 `auto_fix`、`run_version`，或在论文中说明是概念名称。

## 7. PPT检查结果

### 可保留内容

- PPT 共 16 页，结构包括研究背景、系统设计、系统实现、问题与解决方案、系统测试与结果、总结与改进，适合答辩基本节奏。
- 系统架构、核心业务流程、数据库设计、技术栈、工作台、自动修复流程、实验评估等内容与论文主线一致。
- PPT 第 10 页提到测试块识别、类/文件级共享准备代码、修复退化与回滚，能体现选题特色。

### 建议强化内容

- 增加 1 页“现场演示流程”：登录 -> 创建任务 -> 生成代码 -> 运行失败 -> 自动修复 -> 查看版本 diff -> 查看实验报告。
- 增加关键代码或伪代码页，直接展示 `TaskService.auto_fix()` 的循环流程、`RunService.run_version()` 的执行反馈整理逻辑。
- 在实验页补充样本数、模型、数据集、最大轮次、指标口径，不要只展示结论图片。
- 增加部署状态页，说明线上链接是否可用、演示账号、后端 API 是否可访问。

### 建议删减或避免夸大的内容

- 避免把系统表述成能稳定修复所有复杂工程代码；论文也承认目前主要是 Python 函数级、类级、文件级任务。
- 避免强调实时流式进度能力，除非 `stream.py` 已补成真实实现。
- 避免将 Trace/Lesson/Rollback 绝对化为总是提升；本地文件级对比结果显示完整方法并不总是比无 Trace/Lesson 多轮修复更高。
- 避免使用与代码不一致的函数名，如 `task_svc.auto_repair()`、`run_svc.execute()`，建议改成代码实际名称或概念化表述。

## 8. 部署平台检查结果

- 是否访问成功：只能部分确认。带 token 的原始链接返回 `302 Found` 并设置 `eo_token`、`eo_time` cookie；携带 cookie 访问根路径的 HEAD 请求返回 `200 OK`，说明 EdgeOne 静态站点存在。但当前环境中 GET 页面内容多次失败，无法看到实际页面。
- 能看到哪些功能：当前材料不足以判断。未能实际进入登录页、工作台或实验页。
- 是否能支撑答辩展示：当前材料不足以判断。仅凭 HEAD `200 OK` 不能证明系统具备完整演示能力。
- 还缺什么：需要确认线上页面可打开、前端资源可加载、后端 API 地址正确、登录可用、核心流程可跑通、自动修复或至少演示数据可展示。还应提供演示账号、演示任务、备用本地运行方案。

## 9. 是否满足毕业设计基本要求

判断为：B. 基本框架已具备，但还有较明显短板，需要补强。

理由：项目代码、论文、PPT三者总体围绕同一个“代码生成与自修复”系统展开，代码规模和模块结构基本真实，功能也不是普通 CRUD 管理系统，已经具备毕业设计系统原型的主体。但测试支撑、部署可用性、实验数据口径和归档说明存在明显短板，尤其是 Compose SQL 文件名错误、占位测试、实验通过率异常、线上页面未能完整验证，会影响答辩前检查和最终归档可信度。

1. 问题：`docker-compose.yml` 和 README 引用不存在的 `cfix_schema.sql`。应补到：代码/部署/README。建议做法：统一改为 `cfix_db.sql`，重新验证 Compose 数据库初始化。

3. 问题：实验结果存在 200% 通过率。应补到：代码/论文/PPT。建议做法：修复统计脚本，重新生成实验结果并更新论文/PPT。
4. 问题：Trace 指标与论文强调不一致。应补到：代码/论文/PPT。建议做法：解释 trace 非空为 0 的原因，或补齐 trace 采集与展示数据。
5. 问题：线上部署无法完整验证页面内容。应补到：部署/PPT/README。建议做法：确认 token、静态资源、后端 API、演示账号，准备本地备用演示。
6. 问题：PPT/论文部分函数名与代码不一致。应补到：论文/PPT。建议做法：统一为 `auto_fix`、`run_version`、`create_fix_version` 等实际名称，或改成中文概念描述。
7. 问题：缺少用户手册和演示脚本。应补到：README/docs/PPT。建议做法：写一份从零运行和答辩演示步骤，包含账号、题目、测试用例和预期结果。

## 11. 简短结论

当前项目已经具备进入答辩准备阶段的主体基础，但不能按当前状态直接视为完全达标。建议先集中修复部署文件名、测试支撑、实验数据口径和线上演示可用性，再同步调整论文和 PPT 中容易被追问的不一致表述。
